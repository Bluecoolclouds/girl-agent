## [Слои](https://docs.girl-agent.com/docs/developers/architecture/\#%D1%81%D0%BB%D0%BE%D0%B8)

1. **Entry layer** — `cli.ts`, `server.ts`, `headless.ts`. Парсит argv/env, выбирает режим. Никакой логики поведения тут нет.
2. **Runtime** — `engine/runtime.ts`. Главный объект, который держит Telegram-соединение, очередь отложенных ответов, таймеры agenda/daily, EventEmitter для UI. Подробнее: [Runtime](https://docs.girl-agent.com/docs/developers/runtime/).
3. **Behavior layer** — `engine/behavior-tick.ts`. LLM-вызов «что делать с этим сообщением»: ответить/проигнорить/реакция/задержка. Возвращает строгий JSON. Подробнее: [Behavior layer](https://docs.girl-agent.com/docs/developers/behavior-layer/).
4. **Состояния и фоны** — `engine/presence.ts`, `engine/daily-life.ts`, `engine/agenda.ts`, `engine/conflict.ts`, `engine/memory-palace.ts`. Каждый — отдельный «движок» с собственным состоянием и I/O.
5. **Интеграционный слой** — `llm/`, `telegram/`, `mcp/`. Скрывает внешние API за минимальными интерфейсами (`LLMClient`, `TgAdapter`, `McpHandle`).
6. **Storage** — `storage/md.ts`. Один контракт: файлы и папки в `DATA_ROOT/<slug>/...`. Других вариантов хранения нет.
7. **Surface** — `webui/` и `headless.ts`. Превращают события Runtime в HTTP/WS или NDJSON.

   ▼

   │

   │
   ▼

## [Главный data-flow (memory)](https://docs.girl-agent.com/docs/developers/architecture/\#%D0%B3%D0%BB%D0%B0%D0%B2%D0%BD%D1%8B%D0%B9-data-flow-memory)

```
session log/{YYYY-MM-DD}.md (всё, что произошло за день)
   │
   ▼  daily-summarizer закрывает сессию
memory/daily/{YYYY-MM-DD}.md  (саммари дня)
   │
   ▼  memory-palace mineUnminedDailyLogs (асинхронно, фоном)
memory/palace/<wing>/<hall>/<room>/<drawer>.md
                                  (структурированные «комнаты» памяти)
```

Семантически слои разделены так:

- **долгосрочное** — `memory/long-term.md` \+ `memory/facts.md`.
- **эпизодическое** — `memory/episodes/<name>.md`.
- **повседневное** — `memory/daily/<date>.md`.
- **референсное** — `memory/palace/...` (иерархия).

Все читаются в `buildSystemPrompt` (см. `engine/prompt.ts`).

## [События](https://docs.girl-agent.com/docs/developers/architecture/\#%D1%81%D0%BE%D0%B1%D1%8B%D1%82%D0%B8%D1%8F)

`Runtime` — `EventEmitter`. Каждое событие — `RuntimeEvent`:

```
interface RuntimeEvent {
  type: "incoming" | "outgoing" | "ignored" | "score" | "info" | "error";
  text?: string;
  chatId?: number | string;
  reason?: string;
  score?: unknown;
}
```

Все surface'ы (WebUI / headless) — это просто подписчики на `event` \+ ручки на runtime-команды (status/why/wake/reset/...). Логика не зависит от того, кто слушает.

## [Что НЕ является архитектурой](https://docs.girl-agent.com/docs/developers/architecture/\#%D1%87%D1%82%D0%BE-%D0%BD%D0%B5-%D1%8F%D0%B2%D0%BB%D1%8F%D0%B5%D1%82%D1%81%D1%8F-%D0%B0%D1%80%D1%85%D0%B8%D1%82%D0%B5%D0%BA%D1%82%D1%83%D1%80%D0%BE%D0%B9)

- **Нет очереди задач / job runner.** Всё — таймеры на `setTimeout/setInterval` в одном процессе.
- **Нет БД.** Только файлы и `JSON.parse(readFileSync(...))`.
- **Нет сети между модулями.** WebUI и runtime — один процесс, один EventBus.
- **Нет глобального state.** Один профиль = один `Runtime`. Несколько профилей — несколько инстансов, изолированных папкой данных.
- **Нет background-worker'а для сборки персоны.** Persona-pack генерируется один раз на создание профиля; всё остальное — реактивно.

## [Что важно понимать перед тем, как лезть глубже](https://docs.girl-agent.com/docs/developers/architecture/\#%D1%87%D1%82%D0%BE-%D0%B2%D0%B0%D0%B6%D0%BD%D0%BE-%D0%BF%D0%BE%D0%BD%D0%B8%D0%BC%D0%B0%D1%82%D1%8C-%D0%BF%D0%B5%D1%80%D0%B5%D0%B4-%D1%82%D0%B5%D0%BC-%D0%BA%D0%B0%D0%BA-%D0%BB%D0%B5%D0%B7%D1%82%D1%8C-%D0%B3%D0%BB%D1%83%D0%B1%D0%B6%D0%B5)

- **LLM-вызовы сериализованы** глобально (`SerializedLLMClient` в `llm/index.ts`). Это для того, чтобы Anthropic не отвечал 429 на конкурирующие вызовы от одного процесса.
- **Behavior tick — JSON-mode LLM**. Если модель не умеет `response_format`, есть автоматический fallback на `json_object` → `text`.
- **Behavior tick может быть и без LLM** — например, в активном диалоге или при cold-конфликте есть детерминированные ветки. Это сделано чтобы не сжигать токены на каждое «привет».
- **TG-userbot держит сессию в Memory/StringSession**. После логина строка сессии хранится в `config.json` профиля (см. `telegram/userbot.ts`).
- **WebUI — staticfile-сервер + REST**. Никаких SSR, никакого Next. Один React-бандл из `webui/dist`.
